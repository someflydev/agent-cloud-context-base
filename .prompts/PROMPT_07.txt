PART 2 — ARCHETYPE FILES TO AUTHOR
================================================================================

Author the following 12 files in `context/archetypes/`:

2.1  `cloud-function-repo.md`

  Single function with explicit IaC dev/test pair. The narrowest cloud
  archetype. Capabilities: function, iac. Required workflows: add-cloud-
  function, add-iac-stack, add-iac-isolation-pair, add-secret-binding (if
  any secret), add-cloud-smoke-tests, add-cloud-integration-tests.
  Anti-patterns: long-running loops in the handler; mocked managed-service
  in integration test; shared dev/test resource names.

2.2  `multi-function-platform.md`

  Several functions sharing IaC, identity, and a common eventing seam.
  Capabilities: function, eventing, iac, secrets. Required workflows
  add the eventing seam + DLQ + replay. Anti-patterns: pretending the
  platform is one big function; sharing handler code in ways that prevent
  per-function deploy.

2.3  `managed-container-service.md`

  Single Cloud Run / App Runner / Container Apps service. Capabilities:
  container, iac. Required workflows: add-managed-container-service,
  add-cloud-runtime-image, add-iac-stack, add-iac-isolation-pair,
  add-cloud-smoke-tests. Anti-patterns: hosting multiple unrelated
  services in one image to save money; baking secrets into the image.

2.4  `managed-container-multi-service.md`

  Public API + private worker(s) + scheduled job. Capabilities: container,
  eventing, iac, secrets. Workflows add the eventing seam, the VPC/private
  network path, and integration tests across services. Anti-patterns:
  exposing private workers to the public internet; sharing one image
  across services with different runtime needs.

2.5  `k8s-platform-repo.md`

  EKS / GKE / AKS with API + worker + jobs + crons + control plane.
  Capabilities: k8s-api, k8s-worker, k8s-job, k8s-cron, iac. Required
  workflows: add-k8s-workload-role per role, add-helm-or-kustomize-overlay
  for dev/test overlays, add-iac-isolation-pair (cluster + workload
  identity). Anti-patterns: collapsing roles into one Deployment; cluster-
  wide RBAC for tenant code.

2.6  `k8s-multi-tenant-platform.md`

  K8s with namespace-per-tenant, ResourceQuota, LimitRange, NetworkPolicy.
  Capabilities: k8s-api, k8s-worker, k8s-tenancy, iac, secrets.
  Anti-patterns: cross-namespace direct access; tenant-shared secrets;
  unbounded HPAs.

2.7  `cloud-event-pipeline.md`

  Trigger → enrich → persist → fan-out. The dominant shape for ingestion,
  CDC relay, OCR pipelines, etc. Capabilities: function (often with one
  container step for heavier enrichment), eventing, iac, secrets.
  Required workflows: add-replay-and-dlq-handling, add-cloud-storage-
  integration, add-cloud-database-integration, add-cloud-integration-tests.

2.8  `cloud-data-acquisition.md`

  Fetch external sources, archive raw payloads to object storage,
  normalize into managed storage. Mix of functions (HTTP/scheduled
  triggers) and possibly a container worker for heavier parsing.
  Capabilities: function, container, storage, eventing, iac.

2.9  `cloud-rag-pipeline.md`

  Ingest documents, chunk + embed, upsert into a vector store, serve a
  retrieval API. Capabilities: function, container, vector-store, iac,
  secrets. References `context/stacks/observability-otel-cloud.md` and
  the storage stacks.

2.10 `cloud-control-plane.md`

  Thin orchestration layer (function or small container) over heavier
  compute (k8s job, ECS task, Cloud Run job, Container Apps job, Batch).
  Capabilities: function, container, iac, secrets. The plane stays small;
  the compute stays separate.

2.11 `cloud-iac-only-repo.md`

  Platform-team catalog: Terraform modules and/or Pulumi component
  resources, no application code. Capabilities: iac. Required workflows:
  add-iac-stack, add-iac-isolation-pair. Anti-patterns: leaking
  application opinions into module defaults; forcing a specific provider
  region in a module.

2.12 `cloud-multi-provider-experiment.md`

  Comparative AWS / GCP / Azure surfaces for one workload. Capabilities:
  function, container, iac, comparative. Required workflows reference
  every provider's add-*-trigger or add-managed-container-service.
  Anti-patterns: hiding provider-specific behavior under a leaky shared
  abstraction; declaring victory after only one provider passes its gates.

3.1  `context/scenarios/README.md`

  Explain that scenarios are workload taxonomies, not implementation

    - `excellent-cloud-prompts/03--containers--100-excellent-prompts.md`
    - `excellent-cloud-prompts/05--k8s--100-excellent-prompts.md`

  Do not reproduce the full prompt entries. Rephrase the recurring patterns.

3.2  `context/scenarios/function-scenarios.md`

  Distill function-first workload families from the 100 function prompts.
  Required sections:

  - `object-intake-enrichment`: S3/GCS/Blob trigger, metadata validation,
    AI/OCR/moderation/enrichment, managed-store write, fan-out event.
  - `signed-webhook-orchestrator`: HTTP/API trigger, signature verification,
    raw event persistence, dedupe, async handoff, 202 response.
  - `queue-backed-worker`: queue trigger, bounded work item, visibility/
    retry awareness, DLQ, idempotent artifact/status write.
  - `change-event-relay`: stream/change-feed trigger, canonical event
    transform, idempotency marker, downstream bus/topic publish.
  - `identity-or-alert-trigger`: auth/monitoring trigger, ownership lookup,
    audit write, notification or provisioning side effect.
  - `scheduled-report-or-maintenance`: timer/scheduler trigger, bounded
    report or cleanup, deterministic output key, cost/quota guardrails.

  For each family include: typical providers, likely archetype, likely
  manifests, support-service capabilities, canonical example families, and
  validation gates.

3.3  `context/scenarios/container-scenarios.md`

  Distill managed-container middle-band workload families from the 100
  managed-container prompts. Required sections:

  - `public-api-private-worker-job`: public API, private worker/callback
    service, scheduled or event-driven job, managed state, object storage,
    eventing, provider-native secrets.
  - `heavy-runtime-service`: custom image because of Chrome, FFmpeg, GDAL,
    LibreOffice, native image processing, geospatial, or document rendering.
  - `partner-adapter-gateway`: API boundary plus private adapter/dispatcher
    over VPC/VNet/private connectivity, signed archive, retry queue.
  - `multi-container-sidecar`: primary service plus OTel/Envoy/upload
    sidecar in one revision/app when the provider supports it.
  - `job-first-container`: Cloud Run Job / ACA Job / scheduled batch with
    no always-on API, idempotent by run key.

  For each family include likely archetype, provider-specific stack choices,
  support services, IaC isolation defaults, and validation gates.

3.4  `context/scenarios/kubernetes-scenarios.md`

  Distill k8s-worthy workload families from the 100 k8s prompts. Required
  sections:

  - `multi-role-platform`: API + worker + job + cron + optional control
    plane, each separately owned and scaled.
  - `stream-replay-platform`: sustained stream/queue ingest, lag recovery,
    backfill/replay jobs, export crons.
  - `rag-knowledge-mesh`: upload/corpus sync API, extraction/chunking/
    embedding workers, index rebuild jobs, freshness scans.
  - `regulated-document-platform`: secure intake, malware/OCR/classification
    workers, human review, audit/export/retention jobs.
  - `workflow-control-plane`: control API, scheduler/execution workers,
    reconciliation crons, replay of failed steps.
  - `heterogeneous-compute-platform`: CPU/GPU/heavy-memory worker classes
    with explicit node pool and rollout differences.

  For each family include why k8s is warranted, role decomposition,
  provider cluster stack, support services, and validation gates.

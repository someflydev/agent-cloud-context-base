PART 1 — STACK FILE FORMAT
================================================================================

Each stack file in `context/stacks/*.md` follows this shape:

  # <Stack Title>

  <One-paragraph statement of when to load this stack and what it owns.>

  ## Runtime Surface

  - runtime version pin (e.g. python 3.12 / node 20 / go 1.22 / dotnet 8)
  - packaging format (zip, image, source upload, layers)
  - cold-start budget reference (link to doctrine)
  - max execution time + memory
  - supported triggers (provider-native list)

  ## Project Layout

  ```
  <recommended file/dir layout for this stack>
  ```

  ## Handler Skeleton

  ```<lang>
  <minimal handler signature and imports>
  ```

  ## Local Invocation

  ```bash
  <command(s) to invoke locally>
  ```

  ## Idempotency Pattern

  - dedupe key derivation (event-shape-specific)
  - dedupe store binding (DynamoDB / Firestore / Cosmos)
  - replay test command

  ## Identity Binding

  - role/SA/MI shape
  - least-privilege scope rules

  ## Secrets

  - provider secret store + binding pattern (env / SDK / volume)
  - rotation cadence default

  ## Observability

  - structured log shape (JSON fields)
  - trace context propagation source field
  - metrics emission pattern

  ## Validation Gates (cross-reference)

  - <gate-id> from `profile-rules.json`

  ## Anti-Patterns

  - <bullet>

Each stack file 60–150 lines. No frontmatter.

Language policy for this arc:
  - python, typescript, and go are the default cross-cloud function languages
  - dotnet is Azure-specific in workload examples and stack coverage
  - java is not part of the primary scaffold surface
  - ruby and php are excluded from this promptset

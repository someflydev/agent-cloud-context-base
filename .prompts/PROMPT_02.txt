PART 2 — FOUNDATIONAL DOCTRINE FILES TO AUTHOR
================================================================================

Author the following 12 files in `context/doctrine/`. Each filename is
exact. Each must be re-authored from scratch with cloud framing.

2.1  `core-principles.md`

  Smallest useful thing first; v1 over speculative v3. Layer separation
  (doctrine vs workflow vs stack vs archetype vs example vs template vs
  skill vs anchor vs spec vs validation). Real-work focus. Inference over
  internal vocabulary. Canonical patterns over hybrids. Test cloud boundaries
  properly (functions, containers, k8s workloads, managed services).
  Generated repos start lean. Front-facing docs follow implementation. The
  base repo (`accb`) is a base, not a permanent meta-platform.

2.2  `testing-philosophy-cloud.md`

  Default test layers for cloud workloads:
    - unit tests for pure logic (validators, transforms, idempotency-key
      builders)
    - smoke tests that exercise the entry point (function handler, container
      health, k8s pod readiness) with mocked managed-service clients
    - integration tests in one of three lanes (declared per manifest):
        Lane A: local emulator/mock (LocalStack, GCP emulators, Azurite)
        Lane B: ephemeral real (IaC apply against `test` workspace/stack,
                run tests, then destroy) — PREFERRED for medium+ profiles
        Lane C: record/replay for costly third-party calls
  Mocking the managed-service boundary in an integration test is an
  anti-pattern (use Lane A or Lane B). Smoke and integration are different
  tools — never conflate. CI is operator-driven and not added by default.

2.3  `iac-dev-test-isolation.md`

  Dev and test environments must never share state backends, state keys,
  resource names, secret paths, KMS keys, env-var prefixes, or managed
  resource identifiers. Terraform: separate workspaces or per-env state
  backends with disjoint keys. Pulumi: separate stacks per environment with
  isolated state and config secrets. Resource names must differ by
  deterministic suffix (e.g. `<repo>-<env>-<role>`). Test teardown is the
  default; persistent test resources require explicit operator opt-in. The
  `validate_iac_isolation.py` script (PROMPT_17) enforces these rules in CI.

2.4  `trigger-boundary-discipline.md`

  Cloud functions are thin trigger edges, not platforms. A function should
  validate the trigger payload, perform one bounded transformation or
  orchestration step, and hand off to managed services for any heavier work
  (containers for ffmpeg/ML, k8s jobs for long-running compute, Step
  Functions / Workflows / Durable Functions for multi-step orchestration).
  Functions MUST NOT host long-running loops, in-process queues, scheduled
  background tasks (use the platform scheduler), or heavy CPU/GPU work.
  When the trigger boundary becomes blurry, escalate to the
  function-vs-container-vs-k8s doctrine.

2.5  `idempotency-and-replay.md`

  Every cloud-function and event-driven workload must declare an
  idempotency strategy: dedupe key (event ID, object version, message ID,
  business key), an idempotency store (DynamoDB / Firestore / Cosmos /
  Redis), TTL semantics, and the replay safety contract (re-running with
  the same dedupe key never duplicates state). Replay testing is part of
  the integration-test contract: send the same trigger twice and assert
  exactly-once effect at the persistence boundary. DLQ retries inherit this
  contract.

2.6  `function-vs-container-vs-k8s.md`

  Tier selection rule:
    - FUNCTION when the work is one trigger → one short transformation →
      one persistence step, fits within provider time/memory limits, and
      cold-start latency is acceptable.
    - MANAGED CONTAINER when the workload needs custom binaries (ffmpeg,
      GDAL, headless Chrome), 1–N service boundaries with a small set of
      companions (private worker, scheduled job), or steady traffic where
      cold starts hurt. Keep it on Cloud Run / App Runner / Container Apps
      until you need multiple workload classes with separate scaling.
    - KUBERNETES when topology is structural: API + worker fleet + jobs +
      crons + control plane all need separate scaling, recovery, image
      lifecycle, or isolation. K8s costs operational surface — only pay
      that cost when topology demands it.
  When in doubt, start one tier lower and escalate explicitly.

2.7  `identity-and-least-privilege.md`

  Every cloud resource accessed by code must have an explicit identity
  binding: AWS IAM role (Lambda execution role, ECS task role, IRSA for
  EKS), GCP service account (CloudFn SA, Workload Identity for GKE), Azure
  managed identity (system or user-assigned, Workload Identity for AKS).
  Permissions are scoped to the smallest action set that proves the
  workload's trigger contract. Wildcard resources (`*`) require an
  explicit doctrine waiver in the manifest. Identity bindings are part of
  the IaC, not click-ops.

2.8  `cloud-secret-handling.md`

  Secrets live in provider-native secret stores only: AWS Secrets Manager
  / Parameter Store, GCP Secret Manager, Azure Key Vault. They are bound
  to workloads via identity (IAM, SA, MI) — never via copy/paste env vars
  in IaC. Secret values must NEVER appear in source, in `.tfvars`, in
  Pulumi config without `--secret`, in container images, or in logs. Dev
  and test secret paths are disjoint (see iac-dev-test-isolation). Secret
  rotation is a manifest concern; the rotation cadence is declared per
  secret family.

2.9  `observability-cloud-native.md`

  Three pillars for every cloud workload, declared in the manifest and
  exercised in tests: structured logs (JSON, correlation ID, tenant ID,
  trace ID), metrics (provider-native + custom counters for business
  events), traces (OpenTelemetry exported to provider-native backend or a
  vendor). Function workloads require a trace context propagated from the
  trigger payload. Container/k8s workloads require an OTel collector
  sidecar or daemonset. Logs MUST NOT contain secrets, raw PII, or full
  request bodies that include credentials.

2.10 `prompt-first-conventions.md`

  Prompt files live in `.prompts/PROMPT_NN.txt` with strictly monotonic
  numbering. Never renumber, never reuse gaps. Each prompt is
  self-contained for one fresh assistant session: it embeds the shared
  context briefing, declares its boot sequence, lists its parts, names
  its verification commands, and lists its completion gates. Per-session
  state lives in `tmp/PROMPT_NN_checklist.md` (gitignored). Resume and
  completion summaries land in `memory/summaries/PROMPT_NN_*.md`
  (gitignored). The work.py runtime tracks queue, quota, pause/resume,
  and start/done state.

2.11 `canonical-examples.md`

  Examples reduce ambiguity, not create it. One canonical example per
  (provider, runtime, language) cell, with paired Terraform AND at least
  one Pulumi-language equivalent, and dev/test stacks. Examples are real
  runnable code, not prose snippets. Templates are scaffolds; examples
  are completed patterns. When doctrine changes, audit affected examples
  immediately. If no example fits a request, say so explicitly and route
  to the relevant invariant doc plus the closest verified example.
  `examples/catalog.json` is the index; `verification/example_registry.yaml`
  records trust and last-verified state.

2.12 `documentation-timing-discipline.md`

  Derived cloud repos defer substantial root README, root `docs/`, and
  Mermaid architecture diagrams until implementation has real shape to
  describe. Day-one prose is speculative and rots fast. The base repo
  (`accb`) documents itself fully because its job is to be documentation
  for derived repos. Front-facing prose is earned by working code.
  Templates for these docs (`templates/readme/`) are intentionally
  scaffolded as "fill once you have substance" placeholders.

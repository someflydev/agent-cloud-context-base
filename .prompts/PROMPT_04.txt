SHARED CONTEXT BRIEFING — agent-cloud-context-base (accb)
================================================================================

`accb` is a context-first foundation for generating and running
assistant-friendly CLOUD-NATIVE backend repositories: serverless functions
(AWS Lambda, GCP Cloud Functions, Azure Functions), managed containers
(Cloud Run, App Runner, Container Apps), kubernetes platforms (EKS, GKE,
AKS). Modeled on `agent-context-base` (`acb`) but never copies its text.

  - LAYER SEPARATION; ROUTER-FIRST INFERENCE; MANIFEST-DRIVEN BUNDLES;

PART 1 — SPEC NARRATIVE MODULES
================================================================================

Specs are canonical narrative modules used to compose `.accb/specs/*.md` in
derived repos. Every spec file carries YAML frontmatter so the payload
composer can preserve origin metadata.

Author the following in `context/specs/`:

1.1  `context/specs/README.md`

  Short doc explaining: spec layers (product / architecture / agent /
  evolution), the role of frontmatter, and that `scripts/accb_payload.py`
  (PROMPT_16) consumes these to build `.accb/specs/`.

1.2  Frontmatter schema (use in every spec/validation file):

```
---
accb_origin: canonical
accb_source_path: <path relative to repo root>
accb_role: <product | architecture | agent | evolution | validation>
accb_version: 1
---
```

1.3  `context/specs/product/base.md`

  Body: "Baseline Product Intent" — derived repos must make their cloud-
  facing purpose explicit. Every meaningful slice must answer: who or what
  consumes the result; which trigger boundary becomes newly true; what
  behavior would count as broken; which validation result proves the slice
  is real; which cost/quota envelope it consumes.

1.4  `context/specs/architecture/base.md`

  Body: "Baseline Architecture Constraints" — structure stays explicit
  enough that an assistant can identify the changed cloud boundary, the
  owning workload (function/container/k8s role), and the correct validation
  path without broad scanning. Architectural truth must make concrete:
  trigger contracts, identity bindings, secret bindings, persistence seams,
  network seams, IaC dev/test isolation surface, observability surface.

1.5  `context/specs/agent/base.md`

  Body: "Baseline Agent Operating Rules" — agents act autonomously only
  within explicit constraints. Default rhythm: rehydrate local context;
  identify one active boundary; plan validation path before coding;
  implement one reviewable slice; run the relevant checks; update
  continuity artifacts only when the truth changed. Required session
  behavior: re-read `.accb/SESSION_BOOT.md` and `.accb/profile/selection.json`
  at session start; declare provider+runtime+language+IaC tool before
  loading stack packs; do not claim completion until the relevant checks
  actually ran; if proof is blocked, say `blocked`; if work remains
  without proof, say `incomplete`; say `done` only after proof.

1.6  `context/specs/evolution/base.md`

  Body: "Safe Evolution Rules" — new features extend the existing spec/
  validation model instead of bypassing it. Every meaningful change
  preserves or refines: trigger contracts, identity bindings, IaC
  isolation, validation gates, startup order, repo-local operator
  guidance. Spec drift is a real defect because it makes future
  autonomy unsafe.

1.7  Subdirectory READMEs (each a 2–3 line stub)

  `context/specs/product/archetypes/README.md`
  `context/specs/architecture/archetypes/README.md`
  `context/specs/architecture/capabilities/README.md`

2.2  `context/validation/base.md`

  Frontmatter (accb_role: validation), then body: "Baseline Validation
  Contract" — every slice identifies the smallest real proof command set
  for the changed boundary. Minimum expectations:
    - a direct changed-boundary proof (Lane B preferred over Lane A)

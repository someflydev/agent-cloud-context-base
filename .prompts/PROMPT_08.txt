PART 2 — SKILL FILES TO AUTHOR
================================================================================

Author the following 18 skill files in `context/skills/`:

2.1  `cloud-runtime-tier-selection.md`
     Picks function vs managed-container vs k8s using the rules in
     `context/doctrine/function-vs-container-vs-k8s.md`. Procedure:
     identify boundedness, runtime needs, topology, scaling axes; choose
     the lowest tier that fits; document the escalation trigger.

2.2  `provider-selection.md`
     Picks AWS vs GCP vs Azure when the prompt is provider-agnostic.
     Procedure: check repo signals first; check organizational defaults
     (operator declares); check the closest managed-service set to the
     workload; if still tied, ask.

2.3  `language-runtime-selection.md`
     Picks the language for a function or container given provider
     constraints. Procedure: confirm provider runtime support; weigh
     cold-start budget; check existing repo language affinity; pick the
     smallest cognitively coherent runtime.

  2.4  `iac-tool-selection.md`
     Picks Terraform vs Pulumi-{ts,py,go,dotnet}. Procedure: respect
     explicit user request; respect existing repo signals; otherwise apply
     language affinity (Pulumi-language matches the application language
     when possible); treat Pulumi-.NET as an Azure-skewed choice rather
     than a default cross-cloud pick; fall back to Terraform only when no
     Pulumi-language fit exists.

2.5  `trigger-boundary-design.md`
     Keeps function handlers thin. Procedure: enumerate the trigger
     payload's required fields; enumerate the side-effect set; identify
     anything that requires >5s, >256MB, or stateful processing; move
     those out of the function into containers, jobs, or workflows.

2.6  `managed-service-substitution.md`
     Maps a request from one provider's vocabulary to another. Procedure:
     identify the abstract capability (object storage, document store,
     queue, secret store, identity, etc.); look up the provider-native
     equivalent in the relevant stack; flag any semantic gaps explicitly
     (e.g. "Cosmos has change feed; DynamoDB has streams; not identical").

2.7  `secret-binding-strategy.md`
     Picks the secret binding pattern for a workload. Procedure: identify
     the secret family (3rd-party API key, internal cert, KMS-wrapped
     blob); pick provider-native store; bind via identity (IAM/SA/MI);
     declare rotation cadence in the manifest.

2.8  `eventing-seam-selection.md`
     Picks queue vs topic vs bus vs stream for an event-driven seam.
     Procedure: identify ordering needs (FIFO?), fan-out (one-to-many?),
     replay window, batching, DLQ behavior; map to the provider-native
     primitive that matches.

2.9  `k8s-workload-role-decomposition.md`
     Splits a proposed k8s workload into API/worker/job/cron/control-plane
     pieces. Procedure: enumerate runtime classes per piece (latency,
     scaling, lifecycle); separate Deployments per class; document
     cross-piece communication.

2.10 `iac-environment-promotion.md`
     Decides what changes between dev and test stacks/workspaces.
     Procedure: list the per-env axes (state backend, secret path, env
     prefix, resource names, region); confirm dev-only vs shared inputs;
     produce the disjoint config.

2.11 `cloud-integration-test-design.md`
     Picks Lane A (emulator), Lane B (ephemeral real), Lane C (record/
     replay). Procedure: assess managed-service emulator quality; assess
     cost of ephemeral real; assess flakiness of recorded fixtures;
     prefer Lane B for medium+ profiles; combine A+C for cheap scenarios.

2.12 `observability-bundle-assembly.md`
     Picks the smallest useful three-pillar observability bundle.
     Procedure: declare the trace context source; declare the structured-
     log shape; declare the custom metrics for business events; pick the
     OTel exporter target per provider.

2.13 `manifest-selection.md`
     Picks between near-match manifests. Procedure: list candidates;
     score signal strength (lockfile/IaC files/repo signals = strong;
     one-off comments = weak); prefer the narrower scope when scores tie;
     stop and name ambiguity when no manifest dominates.

2.14 `canonical-cloud-example-selection.md`
     Picks the canonical example for a (provider, runtime, language)
     cell. Procedure: filter `examples/catalog.json` by provider, runtime
     tier, language, IaC tool; prefer the most-recently-verified entry;
     reject examples that contradict newer doctrine.

2.15 `scenario-to-profile-selection.md`
     Converts excellent-catalog-style workload prompts into a concrete
     `accb` profile. Procedure: identify the scenario pattern from
     `context/scenarios/scenario-profile-map.yaml`; preserve explicit
     provider/runtime/IaC choices; otherwise select likely manifest,
     archetype, support services, canonical examples, and validation gates
     from the pattern; stop if the scenario implies a different runtime tier
     than the user requested.

2.16 `context-bundle-assembly.md`
     Assembles the first-pass bundle from router output + manifest.
     Procedure: run seven routers; intersect with the chosen manifest's
     required+optional context; apply the complexity-budget cap; emit
     the approved bundle in load order.

2.17 `memory-continuity-discipline.md`
     Decides what goes into MEMORY.md, what goes into a concept artifact,
     and what stays in TASK/SESSION/PLAN. Procedure: durable architectural
     truths → MEMORY.md or concepts/; per-prompt resume/completion →
     summaries/; per-session checklist → tmp/.

2.18 `task-inference.md`
     Translates user free-text into a routed task. Procedure: extract
     verbs (add/build/refactor/fix/deploy); extract nouns (function/
     container/k8s role/secret/event/etc.); pass through alias-catalog;
     route via task-router; if multiple workflows tie, name them and
     ask.

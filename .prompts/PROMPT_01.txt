1. WHAT YOU ARE BUILDING

`agent-cloud-context-base` (`accb`) is a context-first foundation for
generating and running assistant-friendly CLOUD-NATIVE backend repositories.
It is not an app template and not a prompt dump. It is a system for keeping
cloud specs, validation rules, IaC integration discipline, canonical cloud
examples, repo generation, and session startup narrow enough that long
autonomous sessions stay trustworthy.

It is philosophically modeled on `agent-context-base` (`acb`, which targets
local backends, terminal tools, polyglot labs, and Dokku deploys) but is
built FROM THE GROUND UP for the cloud problem space:

  - serverless functions    (AWS Lambda, GCP Cloud Functions, Azure Functions)
  - managed containers      (Cloud Run, App Runner, Container Apps)
  - kubernetes platforms    (EKS, GKE, AKS)

Nothing is copied from `acb`. Every doctrine, anchor, manifest, archetype,
router, stack, workflow, skill, memory artifact, and spec is re-forged with
cloud-native concerns first-class (managed services, triggers, identity,
secrets, IaC dev/test isolation, idempotency, replay, eventing).

2. CORE PHILOSOPHICAL DEBTS TO `acb`

Reproduce these structural ideas (with cloud framing), do not copy text:

  - LAYER SEPARATION: doctrine ≠ workflow ≠ stack ≠ archetype ≠ example
    ≠ template ≠ skill ≠ anchor ≠ spec ≠ validation. Never collapse them.
  - ROUTER-FIRST INFERENCE: users say "add an S3-triggered Lambda" and the
    router picks the smallest useful bundle. No internal vocabulary required.
  - MANIFEST-DRIVEN BUNDLES: yaml manifests enumerate required_context,
    optional_context, preferred_examples, repo_signals, task_hints,
    bootstrap_defaults. Routers/archetypes/stacks reference manifests, not
    raw file lists.
  - CANONICAL EXAMPLES > IMPROVISATION: each example is real, runnable code
    for one preferred pattern per (provider, runtime, language) cell.
  - VALIDATION GATES AS AUTONOMY RAILS: every change to a boundary must name
    its proof path before coding (changed-boundary-proof + provider-specific
    gates). `done` requires proof; otherwise `blocked` or `incomplete`.
  - .accb/ PAYLOAD CONTRACT: a derived repo gets a hidden `.accb/` bundle
    that composes the chosen profile (archetype + primary stack + manifests
    + support services) into `.accb/SESSION_BOOT.md`,
    `.accb/profile/selection.json`, `.accb/specs/AGENT_RULES.md`,
    `.accb/validation/CHECKLIST.md`, `.accb/validation/MATRIX.json`,
    `.accb/validation/COVERAGE.json`, `.accb/INDEX.json`.
  - RUNTIME CONTINUITY: `scripts/work.py` manages `PLAN.md`,
    `context/TASK.md`, `context/SESSION.md`, `context/MEMORY.md`,
    `memory/summaries/PROMPT_XX_*.md`, plus `resume`, `checkpoint`, `next`,
    `pause`, `done`, `start`, `verify`.
  - MEMORY LAYER: `memory/INDEX.md` indexes durable artifacts in
    `memory/concepts/` (architectural truths) and gitignored
    `memory/sessions/` + `memory/summaries/` (per-prompt resume + completion
    summaries).
  - PROMPT-FIRST DISCIPLINE: monotonic `.prompts/PROMPT_NN.txt` files, each
    self-contained for a fresh session, per-session checklist in
    `tmp/PROMPT_NN_checklist.md` (gitignored).
  - CONTEXT COMPLEXITY BUDGET: tiny/small/medium/large/cross-cloud profiles
    cap stacks/archetypes/workflows/examples per first-pass bundle, with
    bounded escalation rules.
  - DOCUMENTATION TIMING DISCIPLINE: defer substantial root README/docs in
    derived repos until implementation has real shape. The base repo itself
    documents fully; derived cloud repos start lean.
  - DOGFOODING: new arcs reuse existing artifacts (e.g. cloud-function
    examples reuse the canonical IaC patterns instead of inventing parallel
    Terraform layouts).

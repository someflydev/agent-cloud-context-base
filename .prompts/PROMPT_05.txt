PART 1 — scripts/work.py
================================================================================

Implement `scripts/work.py` as a single Python 3 file (stdlib only; no
external dependencies) with an argparse dispatcher. Required subcommands:

  init-project         Scaffold runtime files from the *.example.md files.
  resume               Print the Session Context Briefing (details below).
  checkpoint           Snapshot runtime state at a natural boundary.
  next                 Print the next recommended prompt file (if any).
  start PROMPT_NN.txt  Mark a prompt session started (writes a row).
  pause PROMPT_NN.txt  Pause the prompt (reason required).
  done PROMPT_NN.txt   Mark the prompt done (completion summary required).
  recent-commits       Print the last 10 commits with timestamps.
  log-quota            Append a quota-usage row (--assistant, --used-pct-5h).
  verify               Run local structural verification (dispatches to
                       validate_context.py etc. when available).
  budget-report        Score a declared context bundle against the complexity
                       budget (PROMPT_04 `profile-rules.json.startup_features`).
  route-check          Heuristic router output preview from free-text prompt.
  startup-trace        Write a startup-trace record for the current session.
  graft TARGET         Install `accb` runtime continuity into an existing
                       local repo (writes work.py + context/memory rules +
                       a vendored copy of
                       `templates/prompt-first/PROMPT_03.template.txt`
                       as an analysis prompt — NEVER copies doctrine,
                       archetypes, stacks, examples, or validation).

Key behaviors:

  1.1  `resume` prints a "Session Context Briefing" with:
         - git anchor (last commit SHA + subject + timestamp + changed files)
         - runtime-file states (TASK.md, SESSION.md, MEMORY.md, PLAN.md)
           with age, word count, and warnings when stale
         - next-step hint (from TASK.md next-step or plan-review signal)
         - plan-review signal (is PLAN.md stale relative to recent commits?)
         - complexity budget posture (profile guess from active prompt)
         - quota readiness (from the quota log)
         - relevant `memory/summaries/PROMPT_NN_*.md` file (if resuming)

  1.2  `checkpoint` does NOT rewrite TASK.md / SESSION.md automatically; it
       prints a review hint and updates a tiny checkpoint log at
       `work/checkpoint.log` (gitignored under `work/`).

  1.3  `init-project` reads `PLAN.example.md`, `context/TASK.example.md`,
       `context/SESSION.example.md`, `context/MEMORY.example.md` (authored
       in PROMPT_15) and writes the non-example versions only if missing.

  1.4  `graft` is the cloud equivalent of `acb`'s graft command: given a
       target repo path, it writes a minimal `.accb/` discipline layer into
       that repo (work.py copy, `context/memory/` rules, a vendored
       `templates/prompt-first/PROMPT_03.template.txt`
       rendered into the target repo — PROMPT_15 authors the template;
       before then, graft prints "prompt template not yet populated —
       install planned for PROMPT_15").

  1.5  `budget-report`, `route-check`, `startup-trace` gate on the
       `startup_features` block in `context/accb/profile-rules.json`. When
       `enabled: false`, the subcommand prints "feature gated off in
       profile-rules.json" and exits 0.

  1.6  All output is human-readable by default; pass `--json` for machine-
       readable output on any subcommand.

  1.7  Use dataclasses for structured state:
         RuntimeFileSpec, RuntimeFileState, GitAnchor, RepoInspection.

  1.8  Never modify files outside the repo root. Never git commit or push.
       `graft` may write into the target repo; all other commands are
       read-only to non-runtime files.

  1.9  Exit codes: 0 on success; 1 on user error; 2 on environmental error
       (not a git repo, missing runtime file when required, etc.).

Provide `python3 scripts/work.py --help` that lists all subcommands. Each
subcommand has `--help` with a one-paragraph description and flag list.

PART 2 — context/memory/ operating rules
================================================================================

Author the following in `context/memory/`:

2.1  `context/memory/memory-operating-rules.md`

  The durable operating rules for the memory layer:
    - memory/concepts/ holds tracked architectural truths (commit these)
    - memory/sessions/ and memory/summaries/ are gitignored session state
    - memory/INDEX.md indexes concepts/ only (not sessions or summaries)
    - each concept has a short header + frontmatter-free body (50–150 lines)
    - concepts are updated when architectural truth changes, not when
      implementation details change
    - the INDEX.md uses one-line entries with a short hook

2.2  `context/memory/handoff-snapshots.md`

  Rules for writing `memory/summaries/PROMPT_NN_resume.md` and
  `memory/summaries/PROMPT_NN_completion.md`:
    - resume.md captures paused state, the boundary reached, what is safe
      to pick up next, and the full command the next session should run
    - completion.md captures what the prompt delivered, what verification
      passed, and any deferred items
    - both are gitignored; they survive across sessions via `work.py`
    - they never duplicate content that already lives in TASK.md / PLAN.md

2.3  `context/memory/stop-hook-guidance.md`

  When a hook or budget trigger requires stopping mid-session:
    - write `tmp/HANDOFF.md` with the exact resume command
    - write `memory/summaries/PROMPT_NN_resume.md`
    - checkpoint via `work.py checkpoint`
    - state the blocker explicitly in TASK.md (`Blocked by: ...`)

2.4  `context/memory/MEMORY.contract.md`

  Schema contract for `context/MEMORY.md` (the derived-repo continuity
  file). Fields: Repo Purpose; Active Boundary; Non-obvious Invariants;
  Known Gotchas; External References. Limits: 300 lines, 3000 words.
  MEMORY.md is durable repo-local truth. Updates happen at prompt
  boundaries and on architectural changes; not every session.

2.5  `context/memory/MEMORY.template.md`

  A plain template a derived repo can copy into `context/MEMORY.md` on
  init-project. Contains the section headers from Part 2.4 with short
  placeholder prose for each.

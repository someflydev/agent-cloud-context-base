PART 4 — context/router/runtime-router.md
================================================================================

Decides function | managed-container | k8s.

Function signals: thin trigger; one transformation step; provider time/
memory limits acceptable; cold-start budget tolerated; no custom binaries.

Managed-container signals: needs custom binaries (ffmpeg, GDAL, Chrome);
1–N services with a small set of bounded companions; steady traffic where
cold starts hurt; jobs and cron sit alongside an API; sidecars (otel,
envoy) are useful.

K8s signals: API + worker fleet + jobs + crons + control plane; multi-
tenant namespacing; KEDA-style scaling on queue depth; mesh policies;
heavy compute classes (CPU + GPU + memory tiers); rolling-rollout +
rollback contracts that the platform doesn't natively give you.

Reference `context/doctrine/function-vs-container-vs-k8s.md` for
escalation rules. Stop conditions: more than one tier plausible → stop
and ask.

================================================================================

Decides Terraform vs Pulumi-{ts,py,go,dotnet}.

Signals:
  Explicit user request: "terraform" / "pulumi <lang>" → that tool
  Repo signals: `*.tf`, `terraform/` → Terraform
  Repo signals: `Pulumi.yaml`, `Pulumi.dev.yaml`, `__main__.py` with
                pulumi imports, `index.ts` with pulumi imports, `main.go`

  Default by primary language affinity (only when tool is unspecified):
    function/container language python → Pulumi-Python
    function/container language typescript / node → Pulumi-TypeScript
    function/container language go → Pulumi-Go
    function/container language dotnet → Pulumi-DotNet
    function/container language java/kotlin → Terraform
  Multi-language Pulumi parity is a follow-on arc; v1 ships one Pulumi

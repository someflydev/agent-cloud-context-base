PART 1 — STACK FILE FORMAT (k8s variant)
================================================================================

Each k8s stack file follows the container-variant format from PROMPT_10
Part 1 with these k8s-specific adaptations:

  ## Cluster Surface

  - control plane version pin
  - default node pool sizing/instance class guidance
  - networking plugin (VPC CNI / GKE Dataplane V2 / Azure CNI Overlay)
  - ingress story (ALB Ingress / GKE Gateway / Application Gateway / NGINX)
  - identity-to-pod story (IRSA / Workload Identity / Workload Identity for
    AKS)

  ## Workload Surface (per role pack)

  - kind (Deployment / StatefulSet / Job / CronJob)
  - replicas / parallelism / concurrencyPolicy
  - probes (liveness / readiness / startup)
  - resource requests/limits guidance
  - autoscaling (HPA / KEDA / VPA where appropriate)
  - PodDisruptionBudget guidance

  ## Manifests / Charts

  - kustomize base + overlays/dev + overlays/test layout, OR
  - helm chart values.dev.yaml + values.test.yaml layout
  - both layouts are first-class in `accb`; pick per repo preference

  (Networking, Identity, Secrets, Observability, Validation, Anti-Patterns
  remain as in PROMPT_09 Part 1.)

Each k8s stack file 60–150 lines. No frontmatter.

PART 3 — WORKLOAD ROLE PACKS (5 files)
================================================================================

These files are provider-agnostic. They describe the SHAPE of a k8s
workload role and how it composes with the cluster bases above.

  3.1  `k8s-api-workload.md`
       — Deployment + Service + HPA on CPU and/or RPS via custom metric;
         readiness probe gates rollout; PodDisruptionBudget; rolling
         update strategy with maxSurge/maxUnavailable
  3.2  `k8s-worker-workload.md`
       — Deployment + KEDA scaler on queue depth (SQS / Pub/Sub /
         Service Bus / Event Hubs / Kafka); concurrent message processing
         per pod; graceful shutdown signal handling; idempotency contract
         per worker
  3.3  `k8s-job-workload.md`
       — Job per invocation; parallelism + completions explicit;
         backoffLimit; ttlSecondsAfterFinished; activeDeadlineSeconds
  3.4  `k8s-cronjob-workload.md`
       — CronJob with concurrencyPolicy: Forbid (default), startingDead-
         lineSeconds, successfulJobsHistoryLimit, failedJobsHistoryLimit
  3.5  `k8s-control-plane-workload.md`
       — Deployment with leader-election (lease-based) where needed;
         strict resource limits; minimal external surface; long-lived
         informer/reconciler pattern

================================================================================
PART 1 — scripts/accb_payload.py
================================================================================

A library + CLI that composes the `.accb/` payload contents for a derived
repo. Stdlib only (Python 3.10+); pyyaml is acceptable since manifests are
yaml. If YAML support is missing, use the repo-local `.venv-tools` bootstrap
from PROMPT_01 rather than a global install. Do not activate the tools
environment; invoke the needed executable directly from `./.venv-tools/bin/`.
This v1 payload builder only serves scaffoldable single-provider repos. If a
manifest is catalog/reference-only (`generated_repo_target: false`), fail
fast with a clear message and point the operator at the standalone canonical
references from PROMPT_24 instead of trying to coerce null runtime/provider
fields into a runnable repo contract.

Public API (importable from `new_cloud_repo.py`):

  build_payload(
      *,
      archetype: str,
      primary_stack: str,
      primary_language: str,
      provider: str,
      runtime_tier: str,
      iac_tool: str,
      manifest: str,
      support_services: list[str],
      scenario_patterns: list[str] | None = None,
      target_root: pathlib.Path,
      include_startup_features: bool = False,
  ) -> PayloadResult

`PayloadResult` is a dataclass with: written_files (list of paths),
selection (dict matching `.accb/profile/selection.json`), gates_total,
covered_capabilities.

Behavior:

  1.1  Read `context/accb/profile-rules.json`, `context/scenarios/
       scenario-profile-map.yaml`, and the chosen manifest
       (yaml). Validate that `archetype` is in
       `archetype_capabilities`, `manifest` is in `manifest_capabilities`,
       every `support_services` value is in `support_service_capabilities`,
       and every supplied `scenario_patterns` value is in the scenario
       profile map. If `scenario_patterns` is omitted, default to the
       chosen manifest's `scenario_patterns` list.

  1.1b Reject manifests that declare `generated_repo_target: false` (or
       otherwise resolve to a non-scaffoldable contract such as
       `provider: multi`, `runtime_tier: null`, or `primary_stack: none`).
       Raise a clear error that this arc only scaffolds single-provider
       runnable repos and that catalog/reference manifests should use the
       canonical IaC / observability / secrets / eventing materials from
       PROMPT_24 instead.

  1.2  Compute the union of capabilities for this profile:
         capabilities = archetype_capabilities[archetype]
                        ∪ manifest_capabilities[manifest]
                        ∪ flat_union(support_service_capabilities[s] for s in support_services)

  1.3  Compose `.accb/profile/selection.json`:

```json
{
  "schema_version": 1,
  "archetype": "<archetype>",
  "primary_stack": "<stack>",
  "primary_language": "<language>",
  "provider": "<provider>",
  "runtime_tier": "<runtime_tier>",
  "iac_tool": "<iac_tool>",
  "manifest": "<manifest>",
  "scenario_patterns": ["..."],
  "support_services": ["..."],
  "capabilities": ["..."],
  "doctrines": [...default_doctrines from profile-rules...],
  "routers": [...default_routers...],
  "anchors": [...default_anchors...],
  "validation_gates": [
    ...default_validation_gates +
    archetype_validation_gates[archetype] +
    stack_validation_gates.get(primary_stack, []) +
    flat_union(capability_validation_gates[c] for c in capabilities)
  ],
  "startup_features": {
    "budget_report_enabled": <include_startup_features>,
    "startup_trace_enabled": <include_startup_features>,
    "route_check_enabled":   <include_startup_features>
  }
}
```

  Deduplicate `validation_gates` by gate ID before writing
  `selection.json`. If the same gate is contributed by multiple sources,
  keep one entry in `selection.json` and record all contributing sources in
  `MATRIX.json`.

  1.3b Compose `.accb/profile/scenario-patterns.json`: a compact copy of
       the selected pattern records from `scenario-profile-map.yaml`
       (pattern ID, runtime_tier, archetype, likely_manifest for the active
       provider, support_services for the active provider, preferred
       examples). Derived repos verify against this captured profile rather
       than requiring the sibling `excellent-cloud-prompts/` repo.

  1.4  Compose `.accb/specs/{PRODUCT,ARCHITECTURE,AGENT_RULES,EVOLUTION,
       VALIDATION}.md` by concatenating the matching `context/specs/.../base.md`
       and `context/validation/base.md` files (preserving frontmatter)
       with a generated header noting the composition source.

  1.5  Compose `.accb/validation/CHECKLIST.md`: human-readable checklist
       of every gate ID + summary + command_hint.

  1.6  Compose `.accb/validation/MATRIX.json`: machine-readable map of
       gate_id → {category, summary, command_hint, optional command,
       sources: ["default" | "archetype" | "stack" | "capability", ...]}.

  1.7  Compose `.accb/validation/COVERAGE.json`: which capability has
       at least one gate; flag any capability with zero gates.

  1.8  Compose `.accb/SESSION_BOOT.md`: the file an assistant reads on
       session start. Sections: Active Profile (summary), Required Boot
       Reads (in order), Validation Path Reminder, Routing Quick
       Reference (one-liners), Stop Conditions Summary.

  1.9  Vendor `.accb/scripts/accb_payload.py`,
       `.accb/scripts/accb_inspect.py`, and `.accb/scripts/accb_verify.py`
       into the derived repo so inspection and `accb_verify.py --fix`
       work without reaching back to the base repo.

  1.10 Compose `.accb/INDEX.json`: lists every file written, file size,
       SHA-256.

CLI (`python3 scripts/accb_payload.py --help`):

  python3 scripts/accb_payload.py \
    --archetype cloud-function-repo \
    --primary-stack aws-lambda-python \
    --primary-language python \
    --provider aws \
    --runtime-tier function \
    --iac-tool pulumi-python \
    --manifest func-aws-lambda-python \
    --scenario-pattern function.object-intake-enrichment \
    --support-service aws-s3 --support-service aws-dynamodb \
    --support-service aws-secrets-manager \
    --output-dir /tmp/intake

The CLI flag set must match the keyword args of `build_payload`.

PART 2 — scripts/accb_inspect.py
================================================================================

CLI: `python3 scripts/accb_inspect.py [--repo PATH] [--json]`

Reads `<repo>/.accb/profile/selection.json` and prints a tree:

  Profile
  ├── archetype: <name>
  ├── primary_stack: <name> (language: <lang>)
  ├── provider: <provider> | runtime_tier: <tier>
  ├── iac_tool: <tool>
  ├── manifest: <name>
  ├── scenario_patterns
  │   └── [...]
  ├── support_services
  │   ├── <name> → capabilities: [...]
  │   └── ...
  ├── capabilities (union)
  │   └── [...]
  ├── doctrines (loaded by default)
  │   └── [...]
  └── validation_gates
      ├── default        ([n] gates)
      ├── archetype      ([n] gates)
      ├── stack          ([n] gates)
      └── capability     ([n] gates per capability)

`--json` prints the same content as JSON for tooling. Exit 1 if `.accb/`
is missing.

PART 3 — scripts/accb_verify.py
================================================================================

CLI: `python3 scripts/accb_verify.py [--repo PATH]`

Asserts that the derived repo's truth still matches `.accb/`:

  - every file referenced in `.accb/INDEX.json` still exists with the
    recorded SHA-256 (or warn that the assistant has edited the payload
    by hand)
  - every doctrine listed in `.accb/profile/selection.json.doctrines`
    is referenced from at least one boot doc OR appears under
    `.accb/specs/AGENT_RULES.md`
  - every validation gate's executable `command` runs cleanly when
    present (or is annotated as deferred); `command_hint` remains
    human-readable guidance and is not executed blindly
  - every scenario pattern listed in `.accb/profile/selection.json` appears
    in `.accb/profile/scenario-patterns.json`
  - the archetype's repo signals (e.g. presence of `iac/terraform/dev/`
    for an iac_tool of terraform) match the derived repo

Exit codes: 0 clean, 1 mismatch, 2 environmental.

A `--fix` flag re-runs `accb_payload.build_payload` with the saved
selection and overwrites `.accb/` (operator-only; not invoked
automatically).

PART 4 — scripts/new_cloud_repo.py
================================================================================

The user-facing bootstrap entrypoint.

CLI flags (long form first; short forms optional):

  --archetype <name>                 (required)
  --provider <aws|gcp|azure>         (required)
  --runtime-tier <function|managed_container|k8s>  (required)
  --primary-stack <name>             (required)
  --primary-language <name>          (required)
  --iac-tool <terraform|pulumi-typescript|pulumi-python|pulumi-go|pulumi-dotnet>
                                     (required)
  --manifest <name>                  (required)
  --scenario-pattern <pattern-id>     (optional; repeatable; defaults to
                                     the chosen manifest's scenario_patterns)
  --support-service <name>           (zero or more; repeatable)
  --include-dev-test-pair            (default true)
  --include-localstack-or-emulator   (Lane A integration test scaffold)
  --include-ephemeral-real-test      (Lane B integration test scaffold)
  --include-secret-binding-example   (default true)
  --include-observability-bundle     (default true)
  --target-dir <path>                (required)
  --initial-prompt-text <"text">     (optional)
  --initial-prompt-file <path>       (optional; mutually exclusive with above)
  --include-startup-features         (turns on budget/trace/route-check
                                     gates in the derived `.accb/` profile)
  --no-init-git                      (skip `git init` in target)
  --dry-run                          (print plan; write nothing)

Behavior:

  4.1  Validate every flag against `context/accb/profile-rules.json`,
       `context/scenarios/scenario-profile-map.yaml`, and the chosen
       manifest. Any unknown archetype/provider/stack/manifest/scenario/
       service exits 1 with a clear error. If the chosen manifest is
       marked `generated_repo_target: false` (or otherwise implies a
       non-scaffoldable contract), exit 1 with an explicit message that
       `new_cloud_repo.py` only handles single-provider runnable repos in
       v1 and point to the relevant standalone references in PROMPT_24.

  4.2  Resolve the template set: base + chosen language gitignores + iac
       tool templates + function/container/k8s template subset matching
       (provider, runtime_tier, primary_language) + smoke + integration +
       observability templates per the include-* flags + manifest profile
       summary template + prompt-first templates.

  4.3  Render templates (Part 9 of PROMPT_15 placeholder substitution).
       Substitution sources: CLI flags + manifest fields (especially
       env_isolation, iac_layout) + computed values (repo_slug from
       target dir name, selected scenario pattern names, stable random
       suffixes for resource names where documented).

  4.4  Write the derived repo tree:
         <target>/AGENT.md, CLAUDE.md, README.md, .gitignore, PLAN.md
         <target>/.prompts/PROMPT_01.txt
         <target>/.prompts/PROMPT_02.txt
         <target>/iac/terraform/dev/...            (for Terraform manifests)
         <target>/iac/terraform/test/...           (for Terraform manifests)
         <target>/iac/pulumi/<language>/Pulumi.yaml
         <target>/iac/pulumi/<language>/Pulumi.dev.yaml
         <target>/iac/pulumi/<language>/Pulumi.test.yaml
                                                  (for Pulumi manifests)
         <target>/<runtime>/<lang>/...             (handler/Dockerfile/k8s)
         <target>/tests/smoke/...
         <target>/tests/integration/...            (Lane A/Lane B per flags)
         <target>/observability/...
         <target>/scripts/work.py                  (copy of base work.py)
         <target>/.accb/                           (build_payload output,
                                                  including `.accb/scripts/`)
       `PROMPT_01.txt` is the bootstrap prompt and `PROMPT_02.txt` adds the
       first IaC isolation pair.

  4.5  If `--initial-prompt-text` or `--initial-prompt-file` is set,
       write a sentinel `.accb/INITIAL_PROMPT.txt` containing the prompt
       (preserved verbatim for the operator to paste into the first
       session).

  4.6  Print a "Next Steps" footer:
         1. cd <target>
         2. git init -b main && git add . && git commit -m "initial scaffold"
         3. python3 scripts/work.py resume
         4. cat .accb/SESSION_BOOT.md
         5. open .prompts/PROMPT_01.txt in a fresh session

  4.7  Exit 0 on success; 1 on validation; 2 on environmental.
